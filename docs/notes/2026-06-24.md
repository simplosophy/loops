# 2026-06-24

## HLP SDK-only harness control plane

- 删除自研 `loops/loop0` harness、旧 loop0 示例、配置、prompt 和 console script。
- 将项目定位收敛为 HLP SDK：协议对象、`HLPClient`、`HLPHost`、store、event bus 和 adapter contracts。
- 新增 `HarnessAdapter` / `HarnessEvent` / `HarnessCapabilities`，用于把既有 harness 的人工审批、选择、输入和 artifact 事件投影为 HLP 对象。
- 新增 `HLPClient.human_inbox(principal)`，向 Web、IM、CLI 等 host UI 暴露统一的人类待办语义。
- 新增无外部依赖的 `loops-hlp-harness-demo`，验证 existing harness -> HLP checkpoint/artifact/review 的闭环。
- README、HLP spec、架构文档和站点文档同步说明：Loops 不提供 built-in agent harness，执行机制由外部 harness 保留。

## Move HLP implementation into `loops/hlp`

- 将原 `loops/loop2` 下的 HLP 参考实现模块迁移到 `loops/hlp`，让 public namespace 与物理实现目录一致。
- 删除 `loops/loop2` 包路径，不保留兼容 shim，避免继续传播历史分层命名。
- 将协议测试从 `tests/test_loop2_hlp.py` 重命名为 `tests/test_hlp_protocol.py`。
- 将当前架构说明从 `docs/architecture/loop2.md` 重命名为 `docs/architecture/hlp.md`。

## Capability evidence boundary

- 将 HLP core 中的能力引用从 `must_use_capabilities` 收敛为 opaque `ExternalRef`。
- `ExternalRef(kind="capability")` 只作为人类决策、artifact provenance 和 audit replay 的外部证据载体。
- `CapabilityRef` 不再是 HLP core schema 或 conformance 要求；能力发现、schema、鉴权和调用继续由 harness / host / L0 ecosystem 负责。
- 站点和规范同步说明 L0 页面是 capability evidence route，不定义新 capability 协议或 HLP 内建能力模型。

## Codex harness adapter

- 计划新增 `CodexHarnessAdapter`，在现有 Codex CLI prompt/process adapter 基础上实现 `HarnessAdapter`。
- HLP 继续不拥有 Codex execution loop；adapter 只消费 `codex exec --json` 的 JSONL event stream，并投影 human-facing events。
- 端到端闭环覆盖 Codex delegate、checkpoint projection、human resolution、artifact projection、review 和 human inbox。
