# 2026-06-01

## Project rename to loops

- Renamed the public Python package from `loop0` to `loops`.
- Updated distribution metadata, console script, examples, tests, documentation, logging field names, environment variables, and default runtime workspace names to use `loops`.
- Kept legacy `.loop0-*` runtime artifact paths ignored so existing local workspaces do not appear as untracked files after the rename.

## Three-layer architecture draft

- Added `docs/architecture/LOOPS_STACK.md` to define the target `loops/loop0/loop1/loop2` architecture.
- Documented each layer's ownership boundary, state model, extension points, cross-layer event protocol, target package layout, rollout order, and open design questions.
- Clarified that `docs/architecture/OVERVIEW.md` currently describes the single Agent runtime kernel, which maps to target `loop0`.

## Move runtime code into loop0

- Moved the existing single Agent runtime implementation under `loops/loop0/` to match the target layered architecture.
- Kept top-level `loops` imports and legacy submodule paths as compatibility aliases while making `loops.loop0.*` the canonical internal import path.
- Updated tests, examples, README snippets, and architecture directory mapping for the new structure.

## Provider adapter abstraction

- Added a loop0 provider adapter layer with `ProviderModel`, `ProviderOptions`, `ProviderAdapter`, `AdapterBackedProvider`, and adapter registry helpers.
- Refactored `OpenAICompatibleProvider` into a compatibility wrapper backed by `OpenAIChatAdapter` for the `openai-chat` API protocol.
- Updated runtime streaming to accept canonical `text_delta` / `response_finished` provider events while preserving legacy `delta` / `response` compatibility.

## Remove loop0 channels

- Removed the loop0 `Channel` protocol, built-in channel implementations, channel profiles, and top-level `loops.channels` compatibility alias.
- Added explicit loop0 I/O primitives: `InteractionContext` for prompt-visible interaction metadata and `EventSink` for runtime event output.
- Updated `Agent.run()` and runtime streaming so `stream=True` is a direct run option instead of being inferred from channel profile.
- Reworked the sample CLI to keep console interaction outside loop0 and adapt stdin/stdout through `UserInput.interaction_context` and an event sink.
- Rewrote the loop0 architecture overview to make channel/session/storage a loop1 responsibility.

## Generic loop0 CLI runner

- Added `loops-loop0`, a one-shot CLI for running loop0 with either command line flags or a JSON/TOML config file.
- Supported prompt text/files, OpenAI-compatible provider options, shell tool selection, policy options, run/interaction metadata, text/JSON output, and JSONL event export.
- Kept the runner as an embedding host over `UserInput`, `InteractionContext`, and `EventSink` instead of reintroducing channel/session state into loop0.

## Rust migration bootstrap

- Added the initial Rust crate layout with `Cargo.toml`, `src/loop0/*`, and Rust integration tests.
- Ported loop0 configuration parsing, dotenv loading, prompt rendering, OpenAI-compatible non-streaming provider calls, a first tool loop, and basic `shell.run`.
- Added a Rust `loops-loop0` binary as the future replacement target for the Python CLI.
- Documented the staged Python-to-Rust migration plan in `docs/plans/rust-migration.md`.

## Rust loop0 runtime abstraction

- Added Rust `AgentSpec` / `AgentRuntime` so the Rust CLI path runs through a reusable runtime abstraction instead of a config-only helper.
- Added Rust `EventSink` primitives with null and in-memory implementations.
- Added provider streaming output support and runtime conversion to `provider_delta` / `provider_reasoning_delta` events.
- Covered the new runtime/event sink path with a fake provider integration test.
