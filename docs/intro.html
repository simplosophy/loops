AI 协作的 OSI 模型

把 AI 协作
拆成三层协议

MCP 解决了 agent↔工具，A2A 解决了 agent↔agent。
但 agent↔人 在项目/组织级如何协作，至今是空白——Loops 来补这一格。

现状 vs 主张

大多数 harness，都是一根烟囱

今天几乎每一个 AI 框架都在做同一件事：把执行、交互、协同垂直打包进一个单体。 功能齐全，但谁都不敢拆。Loops 主张反过来——各层归各位，用显式协议连接。 协议是各层的"砖"，Loops 是"建筑规范"。

🚫 垂直烟囱

一个项目吞掉三层 · 耦合演进 · 难替换

换一个 provider、加一个审批流，都要动核心 runtime。每一层的进步都被另外两层拖住。

✅ 协议栈分层

三层协议 · 显式契约 · 各自替换

每层选最好的协议实现，跨层只通过显式契约对话。换层不伤筋动骨。

协议栈全景

三层协议，两条契约带

Loops 协议栈是 AI 协作的 OSI 模型。三层各自一个协议，层间靠显式契约咬合。 色带是协议层，青色虚线是契约带， 药丸是每层代表实现——这就是 Loops 视角下的全景。

三层协议详解

每层一个协议，各管一件事

HLP 是 Loops 聚焦的新协议，L1 和 L0 是接入已有协议生态的路由说明—— 核心是补上人机责任闭环，而不是重造 agent/tool 协议。

L0

L0 · 能力路由

Capability protocol routes
说明 HLP 如何引用 MCP、Agent Skills、local tools 或 function calling registry。
做什么：tool.call / skill.invoke / discovery
不做什么：不管 caller 身份、不管会话、不管组织
参考实现：MCP / Skills 协议
形态：路由说明（复用，非新建）
L1

L1 · agent 路由

Agent protocol routes
说明 HLP 如何接入 A2A、ACP、AGNTCY 风格 mesh 或自研 runtime。
做什么：delegate / handoff / discovery
不做什么：不管 agent 内部执行、不管人、不管工具
参考实现：A2A / ACP / AGNTCY
形态：路由说明（复用，非新建）
NEW L2

HLP · 人机协作协议

Human Loop Protocol
定义人↔agent 如何围绕 Task 协作。7 个一等对象，21 个操作。
做什么：Task / Checkpoint / Review / Ownership / Artifact / Ledger / Audit
不做什么：不管 agent 内部、不管 agent 间、不管 UI
参考实现：loops/loop2/（已落地，31 测试）
形态：完整协议规范（Loops 新建 ★）
层间契约

三层怎么缝合

分层之所以不是空谈，靠的是这四个跨层契约。它们是 Loops 栈的接缝规范—— 实现者最容易出错的地方。每个契约有一条铁律。

开源生态地图

那些优秀的项目，各自属于哪一层协议？

把主流开源 harness、协议、编程软件和国内生态放进三层协议栈，会发现： 能在自己层做到极致的项目都很强；而很多最火的产品——尤其 coding agent—— 本身就是一根跨越多层的烟囱。Loops 给它们一张坐标，也指明烟囱的边界。

为什么这样能成立

协议栈不是口号，是硬纪律

分层架构并不新鲜，但 AI 协作比任何场景都需要它——这里每层都在以不同速度变化， 耦合的代价指数级放大。Loops 靠几条不可妥协的纪律把层间边界焊死。

⬇️

依赖只能向下

HLP → L1 route → L0 route。低层永远不能 import 或感知高层。能力实现不该知道 agent 间通信，更不该知道人机协作。

🔌

跨层只走显式契约

CapabilityRef、TaskID 贯穿、Checkpoint→Block、Ownership→Handoff。绝不直接修改邻层内部对象——这是烟囱和协议栈的本质区别。

📍

只前进、可回溯

Task spec、Artifact、Ledger、Review、Audit 全部不可变。要改就新建版本，不就地修改。整个协议可重放、可审计。

结果：换一个 provider 不动 agent 间协议，加一个 IM 不动人机协作，做组织治理不碰能力调用。 每层都能独立替换、独立演进。

别再造烟囱了

选好你要实现的协议层，守住你的状态，暴露干净的契约，
然后和邻层最好的实现组合。这是 Loops 想守住的全部品味。
